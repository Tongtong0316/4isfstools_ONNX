# Windows 人声分离黄金基线

**基线时间**：2026-05-12 23:36（Windows 侧已验证通过）

**适用范围**：仅用于 Windows 人声分离管线的追溯、回归对比与后续修复判定。

---

## 一、这份基线代表什么

这份文档定义的是“Windows 上人声分离已经跑通且被验收”的那一版实现。

后续如果 Windows 分离再次出现以下症状，就应先对照这份基线，而不是继续沿着旧残留目录或旧脚本反复猜测：

- 进度卡在 20%
- CPU 已落下但任务不结束
- 没有生成 `separator_result.json`
- `vocals.wav` / `no_vocals.wav` 未产出
- 安装显示成功，但真正分离时失败
- 旧任务目录残留导致误判

---

## 二、当时确认可用的行为

Windows 分离管线在该时间点满足以下状态：

1. 能够完成一首短音频的分离。
2. 能正确生成结果文件。
3. `separator_result.json` 可以写出。
4. `vocals.wav` 能生成。
5. `no_vocals.wav` 能生成。
6. `success: true` 能回写。

这意味着“安装完成”与“真正能分离”在那一版里是对齐的。

---

## 三、当时的关键修法

### 1. Demucs 子进程输出管道修正

Windows 下分离流程会卡住的核心问题，曾经是 Demucs 子进程的双管道读写导致阻塞。

当时采取的关键修法是：

- 从 `stdout=PIPE, stderr=PIPE`
- 改为 `stdout=PIPE, stderr=STDOUT`

这样可以避免 Windows 下 stdout/stderr 互相堵住，导致分离任务停在固定百分比。

### 2. 进度文件驱动

Windows 分离管线不是只靠一个固定百分比。

它会：

- 先报告起始进度
- 再由 `separator_progress.json` 提供后续进度
- 任务结束后清理进度文件，避免监控线程假死

### 3. 旧任务目录不能当新任务看

Windows 上曾经出现过“旧 `song_*` 目录里残留旧 `separator.py`，让新任务看起来像仍卡 20%”的情况。

因此后续判断必须基于：

- 新建任务目录
- 新生成的 `separator.py`
- 新生成的 `separator_progress.json`
- 新生成的 `separator_result.json`

不能拿旧目录继续误判。

---

## 四、追溯这条基线时应该看什么

如果 Windows 再次出现分离问题，优先检查这些点：

1. `separator.py` 是否还是旧版输出管道写法。
2. 是否真的创建了新的 `song_*` 目录。
3. `separator_progress.json` 是否停止更新。
4. 是否缺少 `separator_result.json`。
5. 任务是不是只完成了安装，但没有真正跑到分离阶段。
6. 运行时依赖是否完整，尤其是分离脚本所需的音频相关依赖。

---

## 五、这份基线对应的维护原则

- Windows 分离管线以后默认以这份文档为准。
- 如果后续修改导致这份基线行为回退，应视为回归。
- macOS 不应再自动继承 Windows 的临时修复，两个平台要分开维护。

---

## 六、建议的后续验证动作

每次 Windows 修改分离管线后，至少做一次以下验证：

1. 用全新音频任务启动。
2. 确认创建新 `song_*` 目录。
3. 确认 `separator.py` 是最新版本。
4. 确认进度不是只停在 20%。
5. 确认结果文件真的落盘。
6. 确认 UI 不只是“看起来在跑”，而是任务确实完成。

---

## 七、结论

这份文档是 Windows 人声分离管线的黄金基线。

后续排查任何 Windows 分离问题，都应先把当前行为与这份基线对照，再决定是：

- 旧任务残留
- 脚本版本不一致
- 进度文件异常
- 进程管道阻塞
- 运行时依赖不完整

不要直接沿着旧 20% 现象继续绕圈。
