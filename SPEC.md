# Macaron Singer 规格书（当前基线）

> 交接与接手摘要见 `HANDOFF.md`，此处保留行为定义与约束。

## 1. 产品目标

- 将本地歌曲处理为可练唱素材（伴奏、歌词、原唱切换）。
- 提供可编辑、可即时保存的歌词时间轴，支持人工微调。
- 当前版本已进入稳定基线阶段，后续改动应优先视为局部修补与体验优化。
- 平台维护策略：macOS 已冻结为短期可用基线；未明确说明平台时，后续开发、调试与交付默认按 Windows 优先处理。
- 歌词来源策略：优先拉取 lrclib.net 的候选结果；右键“搜索匹配歌词”仅打开手动搜索面板，不自动搜；输入关键词后再检索候选并展示列表；后端会对关键词自动做去噪、回退与多轮检索；歌词生成不再依赖本地转写管线。

## 1.1 平台分治约束

- macOS 冻结锚点：`MACOS_FROZEN_2026-05-14.md`
- 平台维护规则：`PLATFORM_MAINTENANCE.md`
- Windows 交接入口：`WINDOWS_AGENT_HANDOFF.md`
- 禁止在 Windows 调试过程中默认重打或修改 macOS 交付包。
- 修改共享代码时应优先使用平台分支或运行时隔离，避免 Windows 修复反向污染 macOS。
- Windows 分离管线验证必须检查新任务目录生成的 `separator.py` 与实际产物，不能用旧 `song_*` 目录残留状态判断新版本。

## 2. 端到端流程

1. 用户拖拽或点击导入音频文件。
2. 歌曲进入 `pending`。
3. 用户在播放列表中右键选择歌词入口：
   - `搜索匹配歌词`：仅打开手动搜索面板；输入关键词后再查询 LRCLib 候选，弹出候选列表供手动选择；后端会自动做去噪与回退检索，随后生成或替换本地歌词文档
4. 完成后如已具备可播放音轨则进入 `ready`。
5. 用户可播放并在歌词面板进行编辑与时间微调。
6. 播放列表可按文件夹组织，支持拖拽分组、折叠/展开、重命名、移动到文件夹与歌词生成入口。

## 3. 状态机

- `pending`：待处理
- `processing`：处理中
- `ready`：可播放
- `error`：处理失败
- `cancelled`：已取消
- `cancelling`：取消中

约束：
- `processing/cancelling` 不允许删除
- `ready` 才允许播放
- `cancelled` 可以重新开始处理
- 首次选歌或重新载入歌曲时，播放器应从 0 开始对齐并播放；同一首歌在 `原唱` / `伴奏` / `人声` 三种模式之间切换时，必须保留当前播放进度，不允许因为切模式回到开头

## 4. 歌词能力规格

### 4.1 数据结构

- 以 `LyricDocument` 为主存储结构（`lyrics.json`）
- 同步生成 `lyrics.lrc` 用于兼容
- 行级字段至少包含：`id/startMs/endMs/text`
- 可选词级字段：`tokens[]`（用于后续单词级高亮/拖拽）
- 输出歌词应保持可编辑、可即时保存，支持后续人工微调与候选替换

### 4.2 交互行为

- 点击歌词行：播放器跳转到该行 `startMs`
- 行内编辑：文本直接改写，触发自动保存
- 时间微调：支持每次 ±100ms
- 当前行高亮：根据播放时间计算 active index
- 播放时自动居中：当前行默认保持在可视区中线附近
- 用户手动滚动让位：播放中手动滚动后，暂停自动居中；短暂静止后恢复自动居中
- 暂停/未播放时不强制回中：歌词可停留在任意滚动位置
- 首尾留白：歌词列表顶部/底部保留动态空白区，保证首行与末行也能自然居中
- 居中实现约束：使用容器滚动位置计算进行精确居中，并隔离“程序滚动”与“用户滚动”事件
- 水平居中约束：歌词文本主内容必须按容器几何中心对齐，左右时间戳/调时按钮不应占用文本中心位

### 4.3 保存策略

- 前端编辑触发去抖保存（当前 400ms）
- 后端落盘 `lyrics.json`，并回写 `lyrics.lrc`
- 当仅有 `lrc` 时，前端需先转换为 `LyricDocument` 再进入统一编辑与保存流程
- 禁止存在“只读 LRC 列表”旁路渲染；所有歌词来源必须走同一 `LyricsPanel`

## 6. UI 约束

- 全局支持拖拽导入
- 底部播放器与内容区保持明确间距，不贴边
- 头部统计与导入按钮需保持可读性（胶囊留白充足）
- 播放器模式切换胶囊（原唱/伴奏）需保证最小高度与横向留白，避免文字拥挤
- 播放主按钮尺寸应控制在不压迫歌词区的范围，保证视觉平衡
- 进度条与控制按钮行应保持明确垂直间距，避免视觉粘连
- 控制区高度必须满足最小值，防止信息行与按钮行挤压导致间距失效
- 时间轴与按钮区之间需保留固定隔离层（结构性间距），禁止仅依赖 margin 微调
- 固定隔离层允许按视觉反馈缩放，但应保持“可辨识分离”与“不产生过大空洞”的平衡
- 固定隔离层支持进一步收缩，优先减少底部空白但不允许时间轴与按钮贴连
- 播放列表文件夹需支持可折叠展示、拖拽分组和右键管理，右键至少包含：搜索匹配歌词（候选列表）、取消处理、重命名、移动到、删除
- 播放列表“新建文件夹”必须有可见输入流程，不能依赖 WebView prompt 这类可能无响应的原生提示框
- 播放列表所有编辑/删除动作应统一使用自定义对话框，不再混用原生 `prompt` / `confirm`
- 新建文件夹弹窗应保持充足内边距、明确 footer 和足够输入宽度，采用纵向分层布局并保证输入框与按钮行明确分离，避免按钮和文本拥挤
- 弹窗样式应优先通过类名固化布局语义，而不是用绝对定位或负 margin 微调按钮位置
- 新建文件夹弹窗底部按钮必须包裹文字形成胶囊，不允许出现文字贴边或按钮过窄
- 提供三按钮单选模式：
  - `原唱`：伴奏轨 + 人声轨 同时播放
  - `伴奏`：仅伴奏轨播放（人声轨静音）
  - `人声`：仅人声轨播放（伴奏轨静音）
- 音量控件应可直接点击调节数值，不仅展示当前音量

## 7. 交互快捷键

- `Space`：播放/暂停切换
- 当焦点位于 `input/textarea/contenteditable` 时不触发全局快捷键，避免干扰歌词编辑

## 8. 非功能要求

- 文档与实现同步：每个小阶段完成后必须更新 `README.md`、`SPEC.md`、`PROGRESS.md`
- 构建可通过：前端 `npm run build`，后端 `cargo check`
- 失败可诊断：处理失败必须可在歌曲项看到错误信息
- 播放器错误分级：`AbortError / MEDIA_ERR_ABORTED` 视为可预期中断，不展示为用户故障
- 播放稳定性约束：切歌、切模式、删除当前歌曲前应先停止并清理旧音频引用，避免叠播和残留状态
- 选歌播放时应优先启动音频，歌词读取应放后台执行，避免把用户点击与 `play()` 请求隔得太远
- 播放键必须在音频引用缺失或失效时提供回补初始化能力，避免出现“点了没反应”
- 原唱模式在首次播放时应先将伴奏轨与人声轨对齐到同一起点，再进行播放，便于观察双轨开头同步效果
- 模式切换约束：`原唱` / `伴奏` / `人声` 仅改变轨道路由与音量，不得重置 `currentTime`
- 取消处理约束：取消必须终止整个进程组，不能只杀父进程；重新处理前需清理残留取消标记
- 运行依赖约束：歌词候选与编辑流程应完全本地可用，不依赖云端写回；缺失候选时应明确提示并允许手动继续编辑本地歌词
- Windows 交付约束（便携包）：优先支持 ZIP 便携交付（解压即可运行，不依赖安装器）；首次运行后可通过「一键安装运行环境」补齐依赖
- Windows 一键补齐约束：当运行时缺失核心依赖时，优先尝试本地离线资源；在线补齐应限制为中国大陆可达软件源策略
- 模型安装约束：最小壳交付场景下，Demucs 与 Whisper base 模型允许通过 `runtime-manifest.json` 提供的大陆可达源在线下载；安装后必须归档到本地 runtime 目录并参与后续健康检测
- runtime manifest 约束：manifest 需支持分平台（macOS / Windows）条目、模型源优先级与 SHA256 字段；安装器按当前平台读取并执行，避免跨平台混用来源
- 进程控制约束（Windows 便携交付前置）：进程终止必须平台隔离；Unix 使用进程组信号，Windows 使用 `taskkill /PID /T`（必要时 `/F`），并避免跨 song 误杀
- 一键安装提示约束：前后端提示必须一致；若 `can_run_core=false`，只能显示“安装未完成 + 缺失项”，不得显示“安装完成，可运行”
